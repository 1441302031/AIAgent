# GSD Project SOP

本项目当前统一使用 **GSD** 作为默认 SOP。

## 核心原则

- 默认按当前仓库结构和当前功能目标推进，不再要求 `superpowers` 工作流。
- 以可执行结果为导向，优先小步迭代、快速验证、及时回归。
- 新功能或行为改动优先先补测试，再补实现。
- 遇到异常先定位根因，再修改，不做盲改。

## 推荐执行顺序

1. 先确认当前目标、范围和成功标准
2. 先阅读相关代码、文档和最近改动
3. 先补最小失败测试或最小复现
4. 用最小改动实现目标
5. 跑局部验证
6. 跑全量回归
7. 更新必要文档

## 文档约定

- 当前有效 SOP 文档放在 `docs/gsd/`
- `docs/specs/` 存放功能规范与分析文档
- 不再新增旧流程目录下的文档
- 历史上的旧流程文档应视为已弃用，不再作为当前流程依据

## 当前项目重点能力

- `library-first + thin CLI`
- `mock / moonshot / deepseek` provider
- streaming 输出
- minimal multi-agent
- 文件上下文预处理

## 本地文件能力原则

- 优先通过自然语言或显式路径从项目根目录解析本地文件
- 支持相对路径和绝对路径
- 默认只读取项目根目录内的安全文本文件
- 写文件能力应单独作为执行层设计，不与读取预处理混在一起
